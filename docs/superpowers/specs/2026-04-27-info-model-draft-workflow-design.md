# Info Model Draft Workflow Design

Дата: 2026-04-27.

## Решение

Основной сценарий создания инфо-модели: `draft из источников -> модерация -> утверждение -> сопоставление параметров -> сопоставление значений -> наполнение товаров -> экспорт`.

Ручное создание модели остается запасным режимом, но не является главным UX-путем.

## Проблема

Сейчас создание инфо-модели, назначение модели категории, сопоставление параметров, конкурентные источники, заполнение товаров и экспорт местами смешаны в одних и тех же рабочих экранах.

Из-за этого пользователь не понимает:

1. где создается структура товара;
2. где система предлагает параметры из источников;
3. где пользователь модерирует и утверждает модель;
4. где параметры PIM сопоставляются с каналами;
5. где значения нормализуются для маркетплейсов;
6. где товар уже просто заполняется по готовым правилам.

## Целевой принцип

Инфо-модель сначала появляется как рабочий draft, затем утверждается как PIM-структура. Только после этого она сопоставляется с внешними каналами.

Нельзя строить базовую инфо-модель прямо внутри category/channel mapping экрана.

## Источники draft-модели

Система собирает кандидаты параметров из:

1. Я.Маркет;
2. Ozon;
3. Excel/import;
4. конкурентные источники `re-store` и `store77`;
5. уже существующие товары выбранной категории;
6. похожие или наследуемые категории каталога.

Каждый кандидат должен хранить provenance:

1. источник;
2. исходное имя параметра;
3. пример значения;
4. сколько раз встретился;
5. confidence score;
6. предложенную группу;
7. предложенный тип значения;
8. возможный словарь значений.

## Workflow

### 1. Категория без модели

Экран `/templates/:categoryId` показывает состояние `Нет модели`.

Главное действие: `Собрать draft-модель`.

Дополнительные действия:

1. создать вручную;
2. скопировать модель из похожей категории;
3. импортировать Excel-шаблон.

### 2. Сбор draft

Пользователь выбирает источники:

1. маркетплейсы;
2. конкуренты;
3. Excel/import;
4. текущие товары категории.

Система запускает сбор и возвращает draft:

1. группы параметров;
2. параметры;
3. типы;
4. обязательность-кандидат;
5. словари;
6. дубли;
7. конфликтующие параметры.

### 3. Модерация draft

Пользователь работает в field builder:

1. принимает параметр;
2. отклоняет параметр;
3. объединяет дубли;
4. переименовывает параметр;
5. меняет группу;
6. задает тип;
7. задает обязательность;
8. привязывает словарь;
9. видит источники и примеры значений.

Пока draft не утвержден, он не должен блокировать товары и экспорт как финальная модель.

### 4. Утверждение модели

После модерации пользователь нажимает `Утвердить модель`.

Результат:

1. модель получает статус `approved`;
2. категория получает активную инфо-модель;
3. товары категории получают список ожидаемых параметров;
4. открывается следующий шаг: сопоставление параметров.

### 5. Сопоставление параметров

Экран `/sources?category=...&tab=params` работает только как mapping-layer.

Он не создает базовую модель. Он берет утвержденную или draft-модель и сопоставляет:

1. параметр PIM -> параметр Я.Маркета;
2. параметр PIM -> параметр Ozon;
3. параметр PIM -> поле конкурентного источника.

### 6. Сопоставление значений

Экран `/sources?category=...&tab=values` нормализует значения.

Пример:

1. PIM: `256 GB`;
2. Я.Маркет: `256 ГБ`;
3. Ozon: `256Gb`;
4. competitor source: `256GB`.

Это отдельный слой. Он не должен менять имя базового PIM-параметра.

### 7. Товары

Карточка товара использует утвержденную модель:

1. показывает обязательные и рекомендуемые параметры;
2. показывает источник каждого значения;
3. показывает конфликт значений;
4. позволяет принять, заменить или вручную ввести значение;
5. показывает как значение будет выглядеть на каждом маркетплейсе.

## Статусы инфо-модели

Минимальный набор статусов:

1. `none` - модели нет;
2. `collecting` - система собирает draft;
3. `draft` - draft собран, идет модерация;
4. `review` - модель готова к утверждению;
5. `approved` - модель утверждена и используется товарами;
6. `needs_update` - появились новые источники или параметры, которые требуют пересмотра.

## Страницы

### `/templates`

Центр инфо-моделей.

Показывает:

1. категории без модели;
2. категории с draft;
3. категории с утвержденной моделью;
4. категории, где модель наследуется;
5. категории, где модель требует обновления.

### `/templates/:categoryId`

Рабочий экран одной категории.

Состояния:

1. `Нет модели`;
2. `Сбор draft`;
3. `Модерация draft`;
4. `Утвержденная модель`;
5. `Модель требует обновления`.

### `/sources?category=...`

Каналы и источники.

Используется после появления модели:

1. category mapping;
2. parameter mapping;
3. value mapping;
4. competitor candidates.

### `/products/:id`

Товарная карточка.

Использует модель, но не редактирует саму модель.

## Backend contracts

На первом этапе можно сделать adapter поверх текущего `templates` слоя.

Минимальные endpoints:

1. `GET /api/info-models`;
2. `GET /api/info-models/{id}`;
3. `GET /api/info-models/by-category/{categoryId}`;
4. `POST /api/info-models/draft-from-sources`;
5. `PUT /api/info-models/{id}/draft`;
6. `POST /api/info-models/{id}/approve`;
7. `POST /api/catalog/categories/{id}/info-model`;
8. `GET /api/info-models/{id}/source-candidates`.

## UI acceptance criteria

1. Пользователь за 5 секунд понимает, есть ли у категории модель.
2. Если модели нет, главное действие одно: `Собрать draft-модель`.
3. Draft показывает источники каждого параметра.
4. Модерация не выглядит как длинная форма на всю страницу.
5. Параметры сгруппированы и редактируются как field builder.
6. Mapping параметров вынесен из создания модели.
7. Mapping значений вынесен из создания модели.
8. Карточка товара показывает trace значения, но не редактирует структуру модели.

## Что не делать

1. Не создавать инфо-модель внутри экрана category/channel mapping.
2. Не делать competitor fields источником истины.
3. Не смешивать value mapping с редактированием PIM-параметров.
4. Не делать одну длинную форму для всех параметров.
5. Не переносить channel-specific обязательность в базовую PIM-модель.

## Следующий implementation scope

Первый кодовый slice после утверждения этого spec:

1. `/templates/:categoryId` получает явные state screens `none / draft / approved`;
2. появляется действие `Собрать draft-модель`;
3. создается компактный draft moderation layout;
4. `/sources` блокирует parameter/value mapping понятным empty state, если у категории нет модели;
5. master-plan обновляется после browser-use проверки.
